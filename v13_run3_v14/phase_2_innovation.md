## THE V14 ARCHITECTURE — COMPLETE DESIGN

### What V14 IS

V14 (the "SIC Deployment Protocol") is a **sequential orchestrator** that deploys a complete initiative stack. It is NOT a new version of the Learning Loop — V13 remains the evaluation engine. V14 WRAPS V13 and adds three more components around it.

**Analogy**: V13 is the quality inspector. V14 is the franchise kit that builds the restaurant, installs the kitchen, hires the inspector, and opens the doors.

### The V14 Stack (Bottom to Top)

```
┌─────────────────────────────────────────────────────────┐
│                    BINGO CITY                            │
│            (What the user sees)                          │
│     Beautiful, gamified, configured by Local North Star  │
│     Powered by: web-db-user scaffold (MySQL/TiDB)       │
├─────────────────────────────────────────────────────────┤
│                 HB1000 OPERATING MANUAL                  │
│            (How the initiative runs)                     │
│     Source of truth, metrics, decision protocols         │
│     Generated by V14, stored in initiative DB            │
├─────────────────────────────────────────────────────────┤
│                 LEARNING LOOP V13                        │
│            (Quality assurance engine)                    │
│     Evaluates, certifies, improves deliverables          │
│     Runs inside V14 as a sub-protocol                   │
├─────────────────────────────────────────────────────────┤
│              CONSTITUTIONAL MEMORY                       │
│            (Governance foundation)                       │
│     Prime North Star, directives, preferences, decisions │
│     Shared across ALL initiatives via GitHub repo        │
└─────────────────────────────────────────────────────────┘
```

### Two-Database Architecture

```
CONSTITUTIONAL DATABASE (GitHub — tim-constitution)
├── CONSTITUTION.md      ← Prime North Star, Noble Cause, Ruby Red
├── DIRECTIVES.md        ← Standing orders (use KEI, cheapest option)
├── PREFERENCES.md       ← Tim's preferences (voice, style, tools)
├── DECISIONS.md         ← Decision history across ALL initiatives
├── KNOWLEDGE.md         ← Index of terms, acronyms, concepts
│                          (PTK = Promises to Keep, NOT Permission to Know)
├── ACTIVE_CONTEXT.md    ← Current state across initiatives
├── knowledge/           ← Deep reference docs
│   ├── pearls-brain.md
│   ├── ptk-system.md
│   ├── blue-seal.md
│   └── ...
└── initiatives/         ← Per-initiative tracking
    ├── ruby-red/
    ├── ptk/
    ├── blue-seal/
    └── ...

INITIATIVE DATABASE (MySQL/TiDB — per Bingo City instance)
├── Local North Star configuration
├── Bingo Card definitions and content
├── Financial data / metrics / source of truth
├── User progress and gamification state
└── Initiative-specific operational data
```

**Relationship**: Constitutional DB INFORMS Initiative DB. Directives constrain what the initiative can do. The initiative DB NEVER contradicts the constitutional DB.

## V14 EXECUTION SEQUENCE — "DROP A V14"

When Tim says "drop a V14 on [initiative]," here's exactly what happens:

### Phase 0: ORIENT (5 minutes)

```
STEP 0.1: Clone constitutional repository
  → gh repo clone tim-constitution
  → Read CONSTITUTION.md (Prime North Star)
  → Read DIRECTIVES.md (standing orders)
  → Read PREFERENCES.md (Tim's preferences)
  → Read KNOWLEDGE.md (terminology, acronyms)

STEP 0.2: Identify initiative
  → Ask: "What is the Local North Star for this initiative?"
  → Ask: "Is this a new initiative or an existing one?"
  → If existing: Read initiatives/[name]/ for context
  → If new: Create initiatives/[name]/ directory

STEP 0.3: Load initiative context
  → Read ACTIVE_CONTEXT.md for cross-initiative state
  → Read DECISIONS.md for relevant past decisions
  → Echo back: "Here's what I know about this initiative: [summary]"
  → Confirm with Tim: "Is this correct? Anything to add?"
```

### Phase 1: ESTABLISH (30-60 minutes)

```
STEP 1.1: Configure Local North Star
  → Document the Local North Star for this initiative
  → Map Local North Star to Bingo Card categories
  → Identify which Trojan Horse approach fits
  → Write to initiatives/[name]/local-north-star.md

STEP 1.2: Scaffold or connect Bingo City
  → If new: Initialize web-db-user scaffold
    → Configure for Local North Star
    → Set up database schema for initiative data
  → If existing: Verify scaffold compatibility
    → Check database schema against V14 requirements
    → Identify gaps and plan upgrades

STEP 1.3: Populate source of truth
  → Identify data sources for this initiative
  → If data exists: Import into initiative database
  → If data needed: Create data collection plan
  → Verify data against constitutional directives
```

### Phase 2: EVALUATE (Variable — depends on deliverable)

```
STEP 2.1: Identify core deliverable
  → What is the main thing this initiative produces?
  → What needs to be evaluated/certified?

STEP 2.2: Run Learning Loop V13
  → Execute full 9-phase V13 on the core deliverable
  → Constitutional memory enforced throughout (Layer 2: Pre-Flight Injection)
  → All directives checked before any recommendation
  → Double Move 37 (Alpha + Omega) active

STEP 2.3: Capture V13 outputs
  → Certification score and report
  → Decisions made → write to DECISIONS.md
  → New knowledge discovered → write to KNOWLEDGE.md
  → Active context updated → write to ACTIVE_CONTEXT.md
```

### Phase 3: BUILD (2-4 hours)

```
STEP 3.1: Generate HB1000 Operating Manual
  → Create initiative-specific operating manual
  → Include: Local North Star, success metrics, decision protocols
  → Include: Team roles, escalation paths, review cadence
  → Store in initiatives/[name]/operating-manual.md
  → Also store key metrics in initiative database

STEP 3.2: Configure Bingo Cards
  → Map Local North Star objectives to Bingo Card squares
  → Each card = one measurable objective or action
  → Configure gamification (points, streaks, achievements)
  → Populate with real data from source of truth

STEP 3.3: Build the beautiful interface
  → Bingo City frontend configured for Local North Star
  → Dashboard shows initiative progress
  → Cards are interactive, trackable, gamified
  → Behind the scenes: full engine running
```

### Phase 4: DEPLOY (30 minutes)

```
STEP 4.1: Verification gate
  → Run Tim's Trust Test on the deployment
  → Verify constitutional directives are enforced
  → Verify source of truth data is accurate
  → Verify Bingo Cards match Local North Star

STEP 4.2: Go live
  → Bingo City accessible to target audience
  → Team members briefed on how to use it
  → Constitutional repository updated with deployment record

STEP 4.3: Handoff
  → Update ACTIVE_CONTEXT.md with deployment state
  → Push all changes to GitHub
  → Next agent (Master Jeeves or Jeeves) can pick up seamlessly
```

## BEFORE/AFTER EXAMPLE (Gap CL3)

### BEFORE V14: Tim starts a new PTK initiative

```
1. Tim opens new chat on Jeeves
2. Tim explains PTK from scratch (20+ minutes of context-setting)
3. Jeeves has no knowledge of Tim's directives, preferences, or past decisions
4. Jeeves recommends ElevenLabs for voice (doesn't know about KEI credits)
5. Jeeves calls PTK "Permission to Know" (doesn't know the real meaning)
6. Tim catches errors, corrects, loses trust
7. No Bingo City, no gamification, no source of truth
8. Work product lives only in this chat — lost when chat ends
9. Next chat starts from zero again
```

### AFTER V14: Tim drops a V14 on PTK initiative

```
1. Tim opens new chat, loads V14 skill
2. V14 clones tim-constitution repo (30 seconds)
3. V14 reads KNOWLEDGE.md — knows PTK = Promises to Keep
4. V14 reads DIRECTIVES.md — knows "use KEI, cheapest option"
5. V14 asks: "Local North Star for this initiative?"
6. Tim says: "Promises to Keep — tracking and nudging promises"
7. V14 scaffolds Bingo City configured for PTK
8. V14 runs V13 on the PTK deliverable
9. V14 generates operating manual, populates source of truth
10. Bingo City goes live — beautiful, gamified, data-driven
11. All decisions written to DECISIONS.md — next chat picks up seamlessly
12. Team members can access via Jeeves with same constitutional foundation
```

**Time saved**: 20+ minutes of context-setting → 30 seconds of repo clone
